# Plan to implement Key Transparency in Samurai.

Create a new binary in `cmd/ktserver/main.go`.
The binary takes 2 arguments as inputs:
- `--addr` to know which IP and port to bind to (defaults `0.0.0.0:3191`)
- `--protocol` which takes one of two values: either `samurai` or `optiks`

This binary works as a bidirectional streaming gRPC server listening to the following gRPC commands:
- `Get(user)`
- `Put(user, key)`
- `GetCommitment`
Here, both `user` and `key` are bytes.

If we are running with `--protocol samurai`, both of these commands should panic with an "Unimplemented" error.

To answer the queries with `--protocol optiks`, do the following:


# OPTIKS protocol


Start by implementing structs:

```go
type OptiksKVPair struct {
    User bytes
    Key  bytes
}

type OptiksServer struct {
    UpdateBuffer    []OptiksKVPair
    RootCommitment  bytes
    rootCommitmentIsDirty bool
    Mpt             MPT // Use go-ethereum's Merkle Patricia Trie (MPT) (see cmd/tools/mptproofs)
    CurrentVersions map[bytes]uint64  // Maps users to their most recent version numbers. If not found defaults to 0
    CurrentKey      map[bytes]bytes   // Maps users to their current key version.
}
```

For all Merkle Patricia Trie, use go-ethereum's package.
During gRPC server initialization, initialize OptiksServer with a blank MPT.
Set RootCommitment to the commitment of the blank MPT and set rootCommitmentIsDirty to false.

### Handling `Put(user, key)`

Simply create a OptiksKVPair with `user` and `key` and store it in updateBuffer.
Set rootCommitmentIsDirty to true.


### Handling `GetCommitment`

If rootCommitmentIsDirty is true, perform the `applyUpdates` operation (described below).

In either case, after this, return RootCommitment as bytes.



### Handling `Get(user)`

If rootCommitmentIsDirty is true, perform the `applyUpdates` operation (described below).

In either case, after this, get the latest version `n` of the key (which might be 0 if the key is not present).

From the MPT, use the get_proof api (or similar methods, look in the docs) for the account generated by concatenating `user` to a byte representation of the uint64 `n+1`.
Since we are using go-ethereum's implementation, this must look like a valid Ethereum account.
This will always be a non-membership proof.

If `n > 0`, then for all `i` in [1, n] (both inclusive) do the following:
- Use the get_proof api (or similar methods, look in the docs) for the account generated by concatenating `user` to a byte representation of the uint64 `i`.


In the end, return the results in the following struct:

```go
type OptiksQueryResult struct {
    Value bytes // Maybe nil if the key doesn't exist
    CurrentVersion uint64
    NextVersionNonMembershipProof []bytes // or use the format MPT returns proofs in
    VersionProofs [][]bytes // Index j (0-based index) in the outer array stores proof for version (j + 1) (to convert it to 1-based)
    // Use the format returned by go-ethereum's proofs if it is not []bytes
}
```


### `applyUpdates`

First, set rootCommitmentIsDirty to false.

Create a new empty slice called `__updates`.
Iterate through each `OptiksKVPair { user, key }` in UpdateBuffer, then do this:

- Insert `user, key` to CurrentKey
- If `user` is not present in CurrentVersions, then initialize to 1, else increment CurrentVersions[`user`]
- Concatenate `user` with the byte-representation of the new value of CurrentVersions[`user`], and convert it to look like an Ethereum account. Store this in a variable called `_account`.
- Convert the `key` to a version suitable to be inserted as a value in the Mpt. Store this in a variable called `_value`.
- Push the tuple `(_account, _value)` in `__updates`


After this loop, clear out UpdateBuffer.

Then batch apply all the updates to the Mpt.
Then use the Mpt APIs to get its root commitment.
Use that to update RootCommitment.
